# i18n - Internationalisation Design

Swedish (`sv`) and English (`en`). Swedish is the default. Language is set per scout group and overridable per user.

## Architecture overview

One source of truth: `api/internal/i18n/messages/` containing `sv.json` and `en.json`. Both the frontend (via Paraglide) and backend (via a thin Go package) read from these files. The Go package embeds them at compile time; Paraglide compiles them at build time. Future developers see one place for all strings regardless of which layer they are working in.

```
api/internal/i18n/
  messages/
    sv.json      <- canonical Swedish strings (embedded by Go, compiled by Paraglide)
    en.json      <- English strings
  i18n.go        <- T(lang, key, vars) and Supported(lang) with //go:embed
web/
  project.inlang/
    settings.json  <- Paraglide config, pathPattern: ../api/internal/i18n/messages/{languageTag}.json
  src/lib/
    paraglide/   <- generated by Paraglide vite plugin at build time (gitignored)
    i18n.ts      <- createI18n(runtime, { exclude: [/\/.*/] }) â€” cookie-based, no URL prefixes
```

The backend and frontend implementations differ (compile-time Paraglide vs runtime Go map lookup) but the data source and key namespace are identical.

## Shared message format

Standard Paraglide-compatible JSON. Keys use underscore-separated namespaces. Paraglide compiles keys to typed JS function names, so dots are not valid â€” underscores only. Variables use `{name}` placeholders.

```json
{
  "article_status_ok": "OK",
  "article_status_reported_usable": "Felrapporterad â€” anvĂ¤ndbar",
  "booking_status_confirmed": "BekrĂ¤ftad",
  "issue_title": "{commercialName} - {severity}"
}
```

Rules:
- Keys are always in English, underscore-separated snake_case.
- Values are in the language of the file.
- Variable names in `{braces}` must be identical across all language files.
- One key = one surface. No reuse of a key for conceptually different strings.
- Prefer reusing an existing key over creating a duplicate with the same value. Before adding a new key, check if the exact string already exists. If the same text appears in multiple places (e.g. "KrĂ¤ver godkĂ¤nnande"), use the same key everywhere. Only create a separate key if the string is likely to diverge across contexts in the future (e.g. `article_status_archived` vs `issue_status_archived` - same text today but different domains).

## Language preference

**Storage (migration 00004):**

```sql
ALTER TABLE group_settings ADD COLUMN default_language text NOT NULL DEFAULT 'sv';
ALTER TABLE users ADD COLUMN language text;  -- NULL = inherit group default
```

**Resolution order:**
1. User's own `users.language` if set and supported
2. Group's `group_settings.default_language`
3. Fallback: `sv`

**Delivery to frontend:**

`/api/v0/me` resolves and returns `language` as part of the user object. `+layout.server.ts` sets the `paraglide_lang` cookie (Paraglide's own cookie name) from `user.language` after each `/me` fetch. The Paraglide handle in `hooks.server.ts` reads this cookie on every request to activate the correct language.

`X-Language` is explicitly stripped from incoming browser requests by the SvelteKit proxy (same pattern as `X-Dev-Role-Override`) â€” identity and locale are injected server-side only.

**Backend-generated strings:**

Handlers read the `paraglide_lang` cookie from the request to determine the language for generated strings. Notification emails (future Phase 3 of the project) will resolve the recipient's language from their `users.language` column at send time.

## Frontend - Paraglide-SvelteKit

**Package:** `@inlang/paraglide-sveltekit` (v0.16.x)

**Language detection:** Cookie-based (`paraglide_lang`), no URL prefixes. `createI18n` is configured with `exclude: [/\/.*/]` which disables URL-based routing entirely â€” no redirects, no `/en/` prefix on any path. Resolution order within Paraglide: `paraglide_lang` cookie â†’ `Accept-Language` header â†’ `sv`.

**Usage:**

```svelte
<script lang="ts">
  import * as m from '$lib/paraglide/messages.js';
</script>

<span>{m.article_status_ok()}</span>
<span>{m.issue_title({ commercialName: 'Sibley 1', severity: m.issue_severity_unusable() })}</span>
```

Paraglide generates one typed function per key. Functions are tree-shaken per language â€” each bundle only includes the active language's strings.

**Note:** `@inlang/paraglide-sveltekit/vite/register-preprocessor` emits a deprecation warning about the `plugin.api.sveltePreprocess` API when running the build. This is an upstream issue in the package and does not affect functionality.

### Migration from labels.ts

`labels.ts` has been removed (Phase 3). All status/label maps now use Paraglide message calls. Non-translatable Tailwind color maps were extracted to `web/src/lib/styles.ts`. A thin `web/src/lib/msg.ts` helper wraps dynamic key lookup (`msg('booking_status_' + status)`) for cases where the key is constructed at runtime.

Before:
```typescript
// labels.ts
export const statusLabels: Record<string, string> = {
  ok: 'OK',
  reported_usable: 'Felrapporterad â€” anvĂ¤ndbar',
};
```

After (inline in components):
```typescript
import * as m from '$lib/paraglide/messages.js';
const label = m[`article_status_${status}`]?.() ?? status;
```

Duplicated inline label maps in `BookingCard.svelte`, `IssueCard.svelte`, etc. are removed when their keys move to messages.

## Backend - Go i18n package

**Package:** `api/internal/i18n`

```go
// T returns the translation for key in lang, substituting {var} placeholders.
// Falls back to 'sv' if key is missing in the requested language,
// then to the key itself if missing entirely.
func T(lang, key string, vars ...map[string]string) string

// Supported returns true if lang is a configured language.
func Supported(lang string) bool
```

Messages are embedded via `//go:embed messages/*.json` â€” no runtime file paths, self-contained binary. Variable substitution uses `strings.ReplaceAll` over `{placeholder}` â€” no ICU parser needed for the small number of backend-generated strings. Both language files are loaded once at first call (sync.Once).

**Usage:**

```go
// In a handler that generates a response string:
lang := "sv"
if c, err := r.Cookie("paraglide_lang"); err == nil && i18n.Supported(c.Value) {
    lang = c.Value
}
title := i18n.T(lang, "issue_title", map[string]string{
    "commercialName": article.CommercialName,
    "severity":       i18n.T(lang, "issue_severity_"+req.Severity),
})
```

**Error messages:** `WriteError` returns English error keys. User-facing errors are translated on the frontend via `translateError()` in `$lib/errors.ts` (see Phase 7). Internal/server errors (5xx, developer-facing validation) are shown as a generic fallback message.

### Phase 7 - API error translation

Goal: user-facing API error messages are translated on the frontend.

**Design:** The API continues to return English error strings as stable keys (`{"error": "article not available"}`). For the 3 dynamic errors that interpolate values, the API adds a `params` object (`{"error": "article_not_available_for_dates", "params": {"name": "Sibley 1"}}`). The frontend maps these keys to translated messages before displaying.

Errors are split into two categories:
- **User-facing** (~30 keys): actions the user triggered and can understand/retry - translated individually via `error_<key>` message keys.
- **Internal** (5xx, developer-facing validation): shown as a generic "Something went wrong" message (`error_generic`). Never translated individually.

**API changes (`api/internal/handler/response.go`):**
- Add `WriteErrorWithParams(w, status, key, params)` that returns `{"error": "key", "params": {...}}`.
- Change the 3 dynamic `fmt.Sprintf` errors in `bookings.go` to use stable keys with params:
  - `"article %s not available for new dates"` â†’ key `article_not_available_for_dates`, params `{"name": "..."}`
  - `"only %d available, requested %d"` â†’ key `insufficient_availability`, params `{"available": "N", "requested": "N"}`
  - `"invalid "+key` in `group_settings.go` â†’ key `invalid_setting`, params `{"field": "..."}`

**Frontend (`web/src/lib/errors.ts`):**
- `translateError(e: unknown): string` - takes a caught error, returns a translated string.
- For `ApiError`: normalizes the error key to a message key (`error_<snake_case_key>`), passes `params` if present. Falls back to `error_generic` for unmapped/5xx errors.
- For non-API errors: returns `error_generic`.

**Message keys (`sv.json` / `en.json`):**
- `error_generic` - generic fallback for internal errors
- ~30 `error_*` keys for user-facing errors (booking conflicts, forbidden, not found, validation, etc.)

**Component updates:**
- All `catch (e: any) { error = e.message }` blocks updated to `catch (e) { error = translateError(e) }`.

## What is explicitly out of scope

- URL-based language routing (`/en/`, `/sv/`) â€” breaks existing links, not needed.
- ICU message format / pluralisation rules â€” not needed for current strings. Add a convention if a plural case arises.
- User-generated content (article names, descriptions, category names) â€” stored as-is, not translated.
- Right-to-left support.
- Translation of stored DB data (issue titles, event descriptions, booking notes).
- Translation management platform (Lokalise, Crowdin) â€” two developer-managed languages in git is sufficient.
